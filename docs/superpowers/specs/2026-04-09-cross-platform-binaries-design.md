# Sub-Project 3: Cross-Platform Native Binaries + CI

**Date:** 2026-04-09
**Status:** Approved
**Goal:** Automate building and publishing node-pty native binaries for all desktop platforms so community users get automatic binary downloads on first terminal open.

## Audience

Same as SP2 â€” Obsidian desktop users installing from the community plugin browser. They install the plugin normally; native binaries download automatically on first use (~2-3MB).

## Scope

1. GitHub Actions workflow to build node-pty for 5 platform/arch targets
2. Zip packaging per platform with checksums
3. Update native-loader to download and extract zips
4. Tag-triggered releases + manual dispatch for testing

Out of scope: signing binaries, auto-update mechanism, building from a node-pty fork.

## Design

### CI Workflow

Single file: `.github/workflows/build-natives.yml`

**Triggers:**
- `push` tag matching `v*` â€” builds binaries and creates a GitHub Release with assets
- `workflow_dispatch` with `electron_version` input (default: `39.6.0`) â€” for testing builds without creating a release

**Matrix strategy:** 5 explicit entries, each defining runner, platform triple, and which files to collect.

**Per-matrix-job steps:**
1. Checkout repo
2. Setup Node 20
3. `npm install node-pty@1.1.0`
4. `npx @electron/rebuild --version $ELECTRON_VERSION --module-dir . --which-module node-pty`
5. Collect platform-specific files into `node-pty-{triple}.zip`
6. Upload zip as workflow artifact

**Release job** (runs after all matrix jobs complete, on `ubuntu-latest`):
1. Download all 5 zip artifacts
2. Generate `checksums.json` â€” SHA256 of each zip, keyed by zip filename:
   ```json
   {
     "node-pty-win32-x64.zip": "abc123...",
     "node-pty-darwin-arm64.zip": "def456...",
     "node-pty-darwin-x64.zip": "...",
     "node-pty-linux-x64.zip": "...",
     "node-pty-linux-arm64.zip": "..."
   }
   ```
3. On tag push: create GitHub Release at that tag, attach all 5 zips + `checksums.json`
4. On manual dispatch: upload artifacts only (no release)

`checksums.json` is also attached as a release asset. For the community plugin browser, `checksums.json` must be present in the repo root so Obsidian includes it when users install. After each release, download the generated `checksums.json` from the release and commit it to the repo. This is a manual step â€” CI does not push commits back to the repo.

### Electron Version

Pinned to 39.6.0 (matches current Obsidian) as workflow default. Parameterized via `workflow_dispatch` input so it can be changed without editing the workflow file. When Obsidian bumps Electron, update the default and re-release.

### worker_threads Patch â€” Not Needed in CI

The production loading path uses a generated shim that calls the native binding directly, bypassing node-pty's JS layer entirely. Combined with `useConpty: false` in `pty-bridge.ts`, the `worker_threads`-dependent `windowsConoutConnection.js` is never loaded. The patch remains in `scripts/dev-install.sh` for local development only.

### Native Loader Updates

`src/terminal/native-loader.ts` changes:

1. **Download URL:** `node-pty-{triple}.node` â†’ `node-pty-{triple}.zip`
2. **Checksum keys:** keyed by zip filename instead of `.node` filename
3. **Post-download:** extract zip to `native/{triple}/` using system commands:
   - Unix: `unzip file.zip -d target/`
   - Windows: `powershell -Command "Expand-Archive -Path file.zip -DestinationPath target/"`
4. **Shim generation:** unchanged â€” shim still references `pty.node` in the same directory
5. **Companion files:** `winpty.dll`, `winpty-agent.exe` (Windows) and `spawn-helper` (macOS) land in the same directory alongside `pty.node` after extraction

No new npm dependencies. System zip tools are universally available on all target platforms.

### Community Plugin Install Flow

1. User installs oterm from Browse Community Plugins â†’ Obsidian downloads `main.js`, `manifest.json`, `styles.css` from GitHub Release
2. User opens terminal for the first time â†’ native-loader downloads platform-specific zip from same GitHub Release, extracts, verifies checksum
3. User sees brief "oterm: downloading terminal binary..." notice
4. All subsequent opens use cached binary â€” no download
